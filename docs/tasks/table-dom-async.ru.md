## Ситуация

Даже с лимитом в 200 строк `generateTableDOM()` создаёт все DOM-элементы синхронно, блокируя UI на время цикла. На слабых машинах или при большом количестве колонок это ощутимо.

## Причина

`generateTableDOM()` — синхронная функция. Цикл по карточкам не отдаёт управление браузеру. Пока все строки не будут созданы и вставлены в DOM, интерфейс не обновляется.

## План

- [ ] **1. Сделать `generateTableDOM` асинхронной**
  - Где: `src/utils/fsrs-table-generator.ts`.
  - Что: `export async function generateTableDOM(...): Promise<HTMLDivElement>`.
  - Зачем: чтобы можно было отдавать управление браузеру между порциями.

- [ ] **2. Разбить создание строк на чанки с отдачей управления**
  - Где: `src/utils/fsrs-table-generator.ts`, цикл `for (const { card, state, isDue } of cardsToShow)`.
  - Что: обрабатывать строки порциями по **50 строк**. После каждой порции — `await new Promise(resolve => setTimeout(resolve, 0))`.
  - Зачем: браузер получает возможность отрисовать уже готовые строки, обновить индикатор загрузки, обработать события.

- [ ] **3. Обновить вызывающие функции — добавить `await`**
  - Где: `src/utils/fsrs-table-generator.ts`, `generateTableDOMFromCards` и `generateTableDOMFromSql`.
  - Что: обещают `Promise`, но внутри синхронно вызывали `generateTableDOM`. После превращения в `async` нужно явно ждать: `const container = await generateTableDOM(...)`.

- [ ] **4. Проверить на хранилище с 10 000 карточек**
  - Собрать бандл (`npm run build`).
  - Открыть заметку с fsrs-table.
  - Убедиться, что таблица появляется постепенно (строка за строкой), а не одним куском.
  - Убедиться, что UI остаётся отзывчивым во время рендера.
