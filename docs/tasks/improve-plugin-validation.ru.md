# Доработка автоматических проверок в `validate-plugin-entry.yml`

Дата: 2025-07-17
Статус: черновик (предложения)

---

## Тип 1. Новые проверки для плагинов

### 1.1 Проверка ESLint/TS-ошибок в коде плагина

**Идея:** Загружать репозиторий плагина, устанавливать зависимости (`npm install`), запускать ESLint и проверять, что нет ошибок выше заданного порога.

**Реализация:**
- После проверки `manifest.json` и релиза — клонировать репозиторий плагина.
- Выполнить `npm install` (или `npm ci`).
- Выполнить `npx eslint src/` (или корневую конфигурацию).
- Если найдены ошибки (error level) — добавить `addError`.
- Если найдены предупреждения (warning level) — добавить `addWarning`.

**Ограничения:**
- Увеличение времени выполнения workflow (npm install может быть долгим).
- Не у всех плагинов есть ESLint; нужна проверка наличия конфига `.eslintrc`, `eslint.config.*` и т.д.
- Ложные срабатывания, если плагин использует нестандартный конфиг.
- Зависимость от внешних ресурсов (npm registry, GitHub).
- Необходимость обработки плагинов на других языках (Python, Rust — см. `isDesktopOnly`).

**Предлагаемые пороги:**
- `0 errors` — обязательно.
- `≤5 warnings` — рекомендуется, иначе предупреждение.

---

### 1.2 Проверка отсутствия `eslint-disable` в коде плагина

**Идея:** Запретить использование `eslint-disable`, `eslint-disable-next-line`, `eslint-disable-line` в коде плагина (кроме особых случаев, например, для WASM-обвязки).

**Реализация:**
- `grep -r "eslint-disable" src/` (или рекурсивно по исходникам).
- Если найден — `addError` с указанием файла и строки.
- Исключения: можно разрешить `eslint-disable` для сгенерированных файлов или специфических ситуаций через файл конфигурации исключений.

**Соображения:**
- Жёсткое правило — многие проекты используют `eslint-disable` легитимно (например, для импорта сборки WASM).
- Лучше сделать это **предупреждением**, а не ошибкой, если есть явная необходимость.
- Или разрешить только с комментарием-пояснением: `// eslint-disable-next-line @reason=...`.

---

### 1.3 Проверка наличия тестов

**Идея:** Проверять, что в репозитории плагина есть хотя бы минимальное тестовое покрытие.

**Реализация:**
- Поиск файлов по шаблонам: `**/*.test.ts`, `**/*.spec.ts`, `**/__tests__/**`, `**/test/**`.
- Проверка конфигов: `vitest.config.ts`, `jest.config.*` в корне репозитория.
- Если тестов нет — `addWarning` (не ошибка, чтобы не блокировать простые плагины).
- Альтернатива мягче: проверять наличие `test` скрипта в `package.json`.

**Ограничения:**
- Не все плагины могут иметь тесты (однофайловые, концептуальные).
- Плагины на Rust + WASM тестируются в Rust, не в JS.
- Порог входа слишком высок для новичков.

**Предложение:** Предупреждение, а не ошибка. Блокировать на усмотрение ревьювера.

---

### 1.4 Проверка на мёртвый код

**Идея:** Запускать TypeScript с флагами `noUnusedLocals`, `noUnusedParameters` и проверять, что нет неиспользуемых переменных/параметров/функций.

**Реализация:**
- Выполнить `npx tsc --noEmit --noUnusedLocals --noUnusedParameters` в клоне репозитория.
- Если есть ошибки — `addError` или `addWarning`.

**Ограничения:**
- Требует наличия `tsconfig.json`.
- Может давать ложные срабатывания на экспортируемые, но не импортируемые функции (если нет `tsconfig` с проверкой файлов).
- Аналогично 1.1 — время выполнения.

---

### 1.5 Проверка отсутствия `console.log` в коде плагина

**Идея:** Запретить `console.log` в production-коде (не в тестах, не в конфигах).

**Реализация:**
- `grep -r "console\.\(log\|warn\|error\)" src/ --include="*.ts"`.
- Если найден не в дебажной секции или без `// DEV` — `addWarning`.
- `console.error` в обработчиках ошибок разрешить.

---

### 1.6 Проверка минимальной версии Obsidian

**Идея:** Проверять, что `minAppVersion` в `manifest.json` не ниже определённого порога (например, `0.15.0` — версия, с которой начался стабильный API плагинов).

**Реализация:**
- После парсинга `manifest.json` сравнить `minAppVersion` через semver.
- Если версия ниже минимальной — `addError` с предложением обновить.

**Порог:** обсуждается. На 2025 год — `0.15.0` или выше.

---

### 1.7 Проверка описания на спам и clickbait

**Идея:** Блокировать описания, которые выглядят как спам (повторяющиеся слова, нечитаемые символы, ALL CAPS, чрезмерное количество эмодзи).

**Реализация:**
- Проверка на повторяющиеся подряд слова.
- Проверка на заглавные буквы >50% текста (кроме акронимов).
- Проверка на количество эмодзи (>3 — предупреждение).
- Проверка на URL в описании — предупреждение.

## Тип 2. Доработки существующих проверок

### 2.1 Ужесточить/ослабить длину описания

**Текущее:** `description.length > 250` — ошибка.

**Предложения:**
- Уменьшить до **200** символов (лучше для UX, короткие описания читаются быстрее).
- Или наоборот, увеличить до **300** и добавить **нижнюю границу** (например, `< 20` символов — ошибка, слишком коротко).
- Добавить поддержку многострочных описаний (сейчас не проверяется перенос строк).

---

### 2.2 Ужесточить правила для `author` и `authorUrl`

**Текущее:** `author` не должен быть email (предупреждение). `authorUrl` не должен указывать на `obsidian.md` и на репозиторий плагина.

**Предложения:**
- **`authorUrl` сделать обязательным** (сейчас опционально).
- Проверять, что `authorUrl` ведёт на валидный URL (не `http://`, не локальный путь, не `file://`).
- Проверять, что `author` не содержит URL.
- **`author` не должен быть пустым или `""`** (сейчас не проверяется явно, только через `hasOwnProperty`).
- Для `fundingUrl` — добавить проверку на валидный URL (не `obsidian.md/pricing`, не пустой).

---

### 2.3 Ужесточить проверку `id` плагина

**Текущее:** только `/^[a-z0-9-_]+$/`, нет `obsidian`, не оканчивается на `plugin`.

**Предложения:**
- **Минимальная длина `id`**: не менее 3 символов.
- **Максимальная длина `id`**: не более 48 символов.
- Запретить **дефис в начале/конце** `id` (например, `-my-plugin-`).
- Запретить **повторяющиеся дефисы** (`my--plugin`).
- Запретить **подчёркивание** в `id` (оставить только дефисы + буквы/цифры) — единый стиль.
- Проверять, что `id` не зарезервирован (список: `obsidian`, `plugin`, `example`, `test`, `sample`, `debug`).

---

### 2.4 Ужесточить проверку `name` плагина

**Текущее:** не содержит `obsidian`, не оканчивается на `plugin`, не начинается на `Obsi`/не оканчивается на `dian`.

**Предложения:**
- **Минимальная длина `name`**: не менее 3 символов.
- **Максимальная длина `name`**: не более 60 символов.
- **Запретить эмодзи** в названии (усложняет поиск и UX).
- **Запретить символы**, кроме букв, цифр, пробелов, дефисов, апострофов — `/^[A-Za-zА-Яа-я0-9\-' ]+$/`.
- Проверять, что `name` не совпадает с `id` (если название такое же, как идентификатор — выглядит странно).
- Проверять, что `name` не является просто числом или символом.

---

### 2.5 Добавить новые обязательные поля в `manifest.json`

**Текущие обязательные:** `id`, `name`, `description`, `author`, `version`, `minAppVersion`, `isDesktopOnly`.

**Предложения — новые обязательные поля:**
- **`authorUrl`** — страница автора (GitHub, сайт). Помогает пользователям и модераторам.
- **`fundingUrl`** — ссылка на поддержку (GitHub Sponsors, Buy Me a Coffee и т.д.). Прозрачность монетизации.
- **`helpUrl`** — ссылка на документацию/поддержку. Сейчас опционально, можно сделать обязательным.
- **`repo`** — дублирование поля из `community-plugins.json` в манифесте (для независимости манифеста).

**Предложения — новые опциональные поля:**
- **`categories`** — массив категорий плагина (по аналогии с темами `modes`).
- **`keywords`** — массив ключевых слов для поиска.
- **`screenshot`** — путь к скриншоту в репозитории (как у тем).
- **`license`** — идентификатор лицензии (SPDX). Сейчас проверяется только наличие LICENSE-файла.
- **`mobileOnly`** — `boolean` (альтернатива `isDesktopOnly` для мобильных плагинов?).
- **`minPluginVersion`** — минимальная версия API плагинов Obsidian (не путать с `minAppVersion`).

---

### 2.6 Ужесточить проверку версии

**Текущее:** только `/^[0-9.]+$/`.

**Предложения:**
- Проверять **строгий semver** (`major.minor.patch`): `/^\d+\.\d+\.\d+$/`.
- Проверять, что версия **не начинается с `v`** (сейчас не проверяется, хотя шаблон PR это упоминает).
- Проверять, что версия в `manifest.json` корня репозитория **совпадает с версией в релизе** (сейчас проверяется только наличие тега, не его значение).
- Проверять, что версия **монотонно возрастает** (сравнение с предыдущей версией в релизах).

---

### 2.7 Ужесточить проверку лицензии

**Текущее:** только наличие LICENSE-файла в репозитории (через `licenses.getForRepo`).

**Предложения:**
- Проверять, что лицензия из **одобренного списка** (MIT, Apache-2.0, GPL-3.0, BSD, MPL-2.0, Unlicense, CC0-1.0).
- Если лицензия не из списка — `addWarning` с предложением использовать стандартную.
- Если лицензия `NOASSERTION` или нераспознана — `addWarning`.
- Альтернатива: добавить поле `license` в `manifest.json` как обязательное (SPDX).

---

### 2.8 Проверка релиза на наличие `styles.css`

**Текущее:** только `main.js` и `manifest.json`.

**Предложение:**
- Если в корне репозитория есть `styles.css` — проверить, что он **не пустой** и **валидный CSS**.
- Если `styles.css` есть в репозитории, но отсутствует в релизе — `addWarning`.
- Если `styles.css` пустой (0 байт) — `addWarning` с предложением удалить.

---

### 2.9 Улучшить валидацию дубликатов

**Текущее:** проверка дубликатов по `id`, `name`, `repo`.

**Предложения:**
- Проверять дубликаты **case-insensitive** (сейчас `p.id === plugin.id` — регистрозависимо).
- Проверять дубликаты **по нормализованному repo** (убрать `.git` в конце, убрать `www.`, убрать trailing slash).
- Проверять **частичное совпадение** `id` с удалёнными плагинами (не только полное совпадение).
- Проверять, что `repo` не указывает на репозиторий, который уже использовался другим плагином (даже если он был переименован).

---

### 2.10 Автоматическая проверка README.md

**Идея:** Проверять, что README существует, содержит минимум информации.

**Реализация:**
- Проверить, что файл `README.md` существует в корне репозитория.
- Проверить минимальную длину (>100 символов?).
- Проверить наличие заголовка с названием плагина.
- Проверить наличие раздела «Установка» или «Installation» (необязательно, но желательно).
- Если README отсутствует — `addError`.

## Тип 3. Инфраструктурные улучшения (опционально)

### 3.1 Кэширование npm зависимостей

Уменьшить время выполнения workflow за счёт кэширования `node_modules` при проверке ESLint/TS.

### 3.2 Параллельное выполнение проверок

Разделить проверки на независимые группы и выполнять параллельно (e.g., проверка репозитория, проверка manifest, проверка релиза, проверка кода).

### 3.3 Уведомления при неудачной валидации

Отправлять автору PR email или GitHub Notification с детальным списком ошибок (сейчас только комментарий к PR).

### 3.4 Retry для внешних вызовов

Добавить retry-логику для запросов к GitHub API (репозиторий, контент, релизы) — сейчас однократный вызов, возможны ложные ошибки при тайм-аутах.
