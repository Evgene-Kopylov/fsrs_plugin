# Задача 1.4. Обработка досрочной остановки

## Описание

Реализовать механизм досрочной остановки прогрессивного сканирования, когда набрано достаточно карточек для заполнения лимита отображения.

## Детали реализации

### Параметр `limitHint`

В `IncrementalCache.startProgressiveScan` передавать `limitHint` как:

```ts
(params.limit || DEFAULT_TABLE_DISPLAY_LIMIT) * 2
```

### Условие остановки

Если в процессе сбора количество собранных карточек (после фильтрации WHERE) достигло `limitHint`, сканирование прерывается, вызывается `onComplete`.

### Фильтрация WHERE

**Проблема:** Фильтрация WHERE должна применяться **до** подсчёта `collectedCards`, иначе `limitHint` будет неточным.

**Варианты решения:**

1. **Желаемый (но дорогой):** Вызывать WASM-функцию `filter_and_sort_cards` на каждом чанке. Отклонён из-за производительности.

2. **Рекомендуемый:** Применить WHERE при добавлении каждой карточки:
   - Для каждой новой карточки проверять через WASM `evaluate_condition`, подходит ли она под SQL условие.
   - Если подходит — добавляем в `collectedCards`, иначе пропускаем.
   - Для проверки нужен доступ к `params.where_condition`. Этот объект должен быть передан в `startProgressiveScan`.

3. **Упрощение на первый раз (минимальная реализация):**
   - Не использовать WHERE для досрочной остановки.
   - Просто собирать все карточки, пока не наберём `limitHint` **любых** карточек (без фильтрации по WHERE).
   - Это менее точно, но проще. Потом можно доработать.

### Функция отмены

- `startProgressiveScan` возвращает функцию `cancel`.
- При вызове `cancel()` флаг `stopRequested = true`, сканирование прекращается.
- В `onunload` рендерера вызывать `progressiveScanCancel?.()`.

## Файлы для изменений

- `src/utils/fsrs/incremental-cache.ts`

## Критерии готовности

- Сканирование останавливается досрочно, когда набрано `limitHint` карточек.
- Функция отмены корректно прерывает сканирование.
- При повторном открытии таблицы кэш уже полный (сканирование не запускается заново).
