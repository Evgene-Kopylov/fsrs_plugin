# Инкрементальный счётчик повторений в кэше WASM

## Проблема

`get_heatmap_data` каждый раз обходит все карточки и все повторения (O(N)), чтобы посчитать `count` для каждой даты. Это горячий путь — рендер вызывается при обновлении, изменении файлов, переоткрытии.

## Решение

Поддерживать `HashMap<date_str, u32>` в глобальном кэше, инкрементально:

- **Инит (скан)** — при `add_or_update_cards` счётчик заполняется. И так уже обходим все повторения, цена — копейки.
- **Add/update** — новые повторения `+=1`, старые вытесненные `-=1` (дельта).
- **Remove** — все повторения карточки `-=1`.
- **Рендер** — `get_heatmap_data` читает готовый `count_by_date`, без обхода карточек. O(371), всегда.

### План

- [ ] Добавить `count_by_date: HashMap<String, u32>` в глобальный кэш
- [ ] Обновлять при `add_or_update_cards`: посчитать дельту повторений, применить
- [ ] Обновлять при `remove_card`: вычесть повторения
- [ ] `get_heatmap_data`: заменить обход карточек на чтение из `count_by_date`
- [ ] `get_heatmap_reviews`: оставить как есть (и так обходит карточки для деталей)

### Файлы

- `wasm-lib/src/cache.rs` — глобальный кэш и все три функции

### Обоснование

Это не перекладывание задержки, а устранение повторной работы из горячего пути. Сейчас O(всех повторений) платится при **каждом** рендере (обновление настроек, изменение файла, переоткрытие). С инкрементальным счётчиком:

- Инит кэша и так обходит все карточки — `+=1` в HashMap почти бесплатно.
- Add/update/remove — O(1) на повторение (дельта).
- Рендер — O(371) всегда, без обхода карточек.

При повторных рендерах разница ещё заметнее.

### Критерий готовности

- `get_heatmap_data` не обходит карточки — только O(371)
- Счётчик корректен после add/update/remove
- Рендер < 0.09 с
