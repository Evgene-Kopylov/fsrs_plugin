# Ленивое сканирование хранилища

## Проблема

При загрузке плагин сразу сканирует все 45 000+ файлов хранилища:
парсит frontmatter, вычисляет состояния и заполняет WASM-кэш.
Это занимает ~18 секунд и нагружает диск/процессор, даже если
пользователь просто открыл Obsidian без единой fsrs-таблицы.

## Какие функции требуют кэш

Кэш (полный обход всех файлов) нужен только когда нужно
фильтровать/сортировать карточки по хранилищу:

1. **Таблица `fsrs-table`** — `renderContent()` вызывает
   `cache.query()` с WHERE/ORDER BY/LIMIT.
2. **Команда поиска карточек** — ищет карточки, готовые к повторению.

**Кнопка `fsrs-review-button`** не требует кэша: она читает и
парсит frontmatter только одного файла через WASM
(`extract_fsrs_from_frontmatter_wrapped`). Ей достаточно, чтобы
WASM был инициализирован (`isWasmReady()` / `onWasmReady`).

## Решение

Перенести `performCacheScan()` из `onLayoutReady` в момент
**первого реального запроса к кэшу**.

1. **Убрать** вызов `performCacheScan()` из `onLayoutReady`.
2. **Вызывать** `performCacheScan()` только при:
   - первой регистрации `FsrsTableRenderer` (когда появляется
     блок `fsrs-table`),
   - первой команде поиска карточек.
3. Пока сканирование не завершено — таблица показывает loading.
4. После завершения — уведомить всех ожидающих
   (механизм, аналогичный `onWasmReady`).

Кнопка `fsrs-review-button` не трогает кэш — она работает
через `onWasmReady` (уже реализовано).

## Детали

- Сканирование должно быть **однократным**: если один запрос уже
  запустил сканирование, остальные должны дождаться его завершения,
  а не запускать повторно.
- Прогрессивное сканирование с досрочной остановкой по лимиту
  (2 × LIMIT) — уже заложено концептуально, но в текущем коде
  не реализовано; можно сделать на этом же этапе.
- Если сканирование уже выполняется, а пришёл новый запрос —
  просто подписаться на промис завершения.
- Файловые события (`modify`, `delete`, `rename`) продолжают
  инкрементально обновлять кэш — их не трогаем.
- После первого завершения сканирования кэш остаётся в памяти
  до перезагрузки плагина.

## План

1. Убрать `performCacheScan()` из `onLayoutReady`.
2. В `FsrsPlugin` добавить:
   - `private scanPromise: Promise<void> | null = null`
   - `private ensureCacheScanned(): Promise<void>` — запускает
     сканирование, если ещё не запущено, и возвращает промис,
     который резолвится по окончании.
3. Вызывать `ensureCacheScanned()` в:
   - `FsrsTableRenderer.renderContent()` (перед первым запросом
     к `cache.query()`)
   - `findCardsForReview()` в команде
4. `notifyFsrsTableRenderers()` в конце `performCacheScan()`
   заменится на разрешение `scanPromise`.

## Критерии готовности

- [ ] При чистом старте без `fsrs-table` сканирование **не запускается**.
- [ ] При открытии заметки с кнопкой `fsrs-review-button` сканирование
      не запускается (кнопка работает без кэша).
- [ ] При открытии заметки с таблицей — сканирование запускается,
      таблица заполняется после его завершения.
- [ ] Повторный вызов `ensureCacheScanned()` во время сканирования
      не запускает второй проход.
- [ ] После сканирования все последующие запросы работают
      из кэша (без повторного сканирования).
- [ ] Команда поиска карточек запускает сканирование, если его
      ещё не было.
