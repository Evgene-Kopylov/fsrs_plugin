# FSRS-плагин для Obsidian: SQL-подобные запросы к карточкам, Rust/WASM и столетний интервал

Интервальное повторение в Obsidian — тема, где большинство плагинов останавливаются на алгоритме SM-2 образца 1987 года. Я пошёл дальше: реализовал FSRS (Free Spaced Repetition Scheduler), написал вычислительное ядро на Rust (используя готовую реализацию алгоритма библиотекой `rs-fsrs`) и прикрутил SQL-подобный язык для фильтрации карточек. В статье — архитектура с WebAssembly, собственный парсер и неочевидные решения.

## Зачем четвёртый плагин для повторений?

На момент написания в Obsidian есть три популярных решения: `obsidian-spaced-repetition`, `obsidian-recall` и `obsidian-review`. Все используют SM-2 — алгоритм которому скоро 40 лет. Он работает, но требует примерно на 30% больше повторений чем FSRS при том же уровне запоминания.

Главные недостатки SM-2, и что это вообще такое:
- это повторение с нарастающими интервалами; если пропустил — начинаешь сначала. Все расчёты на клочке бумаги. Это и есть SM-2.
- Одинаковый интервал для материала любой сложности
- Не обрабатывает пропуски — «сброс» прогресса после перерыва
- Нет понятия извлекаемости (retrievability) — вероятности вспомнить карточку сейчас.

FSRS это шаг вперед. Но его нет в Obsidian. Точнее — не было.

## Как устроен FSRS в двух словах

FSRS оперирует DSR-моделью из трёх параметров:

1. **Difficulty (сложность)** — насколько труден материал. Статическая характеристика, почти не меняется со временем. Диапазон: 0–10
2. **Stability (стабильность)** — прочность запоминания в днях. Через столько дней извлекаемость упадёт до 90%.  
   > При высокой стабильности и низкой желаемой частоте повторений следующий интервал может измеряться десятилетиями — отсюда «столетний интервал» в заголовке.
3. **Retrievability (извлекаемость)** — динамическая вероятность вспомнить карточку прямо сейчас. Вычисляется из стабильности и времени с последнего повторения

После каждого ответа (Again / Hard / Good / Easy) алгоритм пересчитывает сложность и стабильность. Извлекаемость меняется непрерывно — каждую секунду она немного падает.

Алгоритм использует 21 параметр, подобранный машинным обучением на миллионах реальных повторений. Пользователь задаёт желаемый уровень запоминания (request retention) — обычно 90%. 

Алгоритм ПОКАЗЫВАЕТ насколько забыт материал 😮.

## Архитектура: почему Rust и WebAssembly

Плагин Obsidian — это JavaScript. Но FSRS требует точных вычислений с плавающей точкой на каждом повторении. Можно ли написать на TypeScript? Можно. И многие так делают.

Я выбрал Rust
~~из чистой незамутненной любви~~
~~потому что он простой~~ 
~~потому важен процесс а не результат~~ 
по трём причинам:

1. **Производительность** — WASM работает на порядок быстрее JS на численных вычислениях
2. **Экосистема** — крейт `rs-fsrs` от сообщества open-spaced-repetition с эталонной реализацией FSRS
3. **Безопасность типов** — в Rust невозможно перепутать `difficulty` и `stability`, а большая кодовая база на чистом TypeScript без компилятора быстро превращается в месиво

Разделение ответственности:

```
TypeScript                  Rust/WASM
─────────                   ─────────
• Obsidian API              • FSRS-вычисления
• UI / рендеринг            • Парсинг SQL-подобного синтаксиса
• Файловая система          • Парсинг YAML/JSON
• Жизненный цикл плагина    • Фильтрация и сортировка
• Кнопки, модалки           • Кэш карточек
```

WASM → TS: «отфильтрованные карточки с состояниями». Поток: Obsidian → TS-обвязка → WASM-ядро (кэш + FSRS + парсер) → обратно. -->

TypeScript — тонкая обвязка над Obsidian API. Вся логика — в WASM. Механизм прост: TS читает файлы, передаёт данные в WASM, получает результат, отображает.

## Производительность: граница WASM ↔ JS

Не будет ли узким местом передача тысяч карточек из Rust в JavaScript?

Да, если делать наивно — будет. Каждый вызов функции из WASM возвращает данные, которые нужно сериализовать через `serde_wasm_bindgen`, а это копирование каждого поля каждой карточки. Для 1000 строк таблицы это может занять 30-50 мс — заметно для интерфейса.

### Минимизация копирования

Я использую три приёма:

1. **Кэш внутри WASM** — фильтрация и сортировка выполняются там же, в Rust. Через границу передаётся только результирующий список карточек (те, которые попали в `LIMIT` 20-200 строк), а не все 10 000.
> на этом можно закончить. 200 строк это ни о чем.

2. **Инкрементальные обновления** — при ответе на карточку пересчитывается только одна запись, а не вся таблица. Через границу идёт микроскопический пакет.
> а вот это прикольно

### Цифры

Замеры на реальном хранилище (105 606 `.md` файлов, 4 964 карточки FSRS):

- **Сканирование хранилища** (ленивая загрузка — только при первом открытии `fsrs-table` за сессию):  
  Чтение всех `.md`, парсинг frontmatter, фильтрация игнор-списка, отбрасывание файлов без frontmatter и без FSRS-полей, построение кэша — **≈3.1 с**.  
  *Для типичного хранилища (5–10 тыс. файлов) — 0.3–0.6 с.*

- **Загрузка таблицы** (после того как кэш построен):  
  Полный обход всех карточек, фильтрация, сортировка по умолчанию (`retrievability ASC`), сериализация 20 строк через границу WASM→JS, рендеринг в Obsidian — **0.06 с** (60 мс).

- **Обновление одной карточки** после нажатия кнопки (Again/Hard/Good/Easy):  
  Пересчёт параметров, обновление кэша, перерисовка активной таблицы — **менее 1 мс**.

На практике единственное заметное ожидание - при первом открытии таблицы в сессии (и только для очень больших хранилищ). Все остальные действия выполняются быстрее, чем глаз успевает моргнуть.

### Что не так (о компромиссах)

`LIMIT` в текущей реализации влияет **только на количество возвращаемых строк**, но не прерывает обработку набора данных. Алгоритм:

1. Полный обход всего кэша карточек (например, 50 000 записей)
2. Фильтрация по `WHERE`
3. **Сортировка по умолчанию** — если пользователь не указал `ORDER BY`, плагин автоматически добавляет `ORDER BY retrievability ASC` (самые забытые карточки сверху). Это критично для интервального повторения — пользователь всегда видит в первую очередь то, что нужно повторять.
4. Обрезка до `LIMIT` → возврат в JS

**Почему так, а не досрочная остановка как в настоящих БД?**

Чтобы гарантированно получить первые N строк по извлекаемости, 
всё равно нужно оценить все карточки — нельзя заранее знать, 
что в конце списка не окажется карточка с ещё более низкой retrievability. 
Это свойство глобальной сортировки. Настоящие БД с индексом по полю
`retrievability` могли бы остановиться досрочно, 
но у нас нет индексов — это плата за простоту.

**Компромисс принят сознательно:** размер хранилища реального 
пользователя редко превышает 5000–10000 карточек, 
полный обход + сортировка занимают 5–10 мс. Для пользователя 
задержка незаметна, зато поведение предсказуемо — никакая карточка 
не будет пропущена из-за магического `LIMIT` до сортировки.

## Кэш в WASM вместо локального стейта

Обычный подход для Obsidian-плагина: загрузить данные при старте, хранить в локальной переменной TypeScript, обновлять по событиям файловой системы. 

Я сделал иначе: весь кэш (`HashMap<filePath, CachedCard>`) живёт внутри WASM как глобальная переменная. Плагин не хранит состояния в TypeScript вообще.

Почему:

- **Единый источник истины** — нет рассинхрона между JS-стейтом и WASM-вычислениями
- **Быстрые запросы** — фильтрация/сортировка идут там же где данные, без копирования через границу TS/WASM
- **Инкрементальное обновление** — при изменении файла TS посылает WASM точечную команду: «обнови эту карточку» или «удали эту». Не перестраивает всё

**Безопасность кэша.** Проблема которой не возникло. WASM-модуль компилируется в отдельный `.wasm` файл и подключается как часть плагина. При каждой загрузке плагина (включая горячую перезагрузку после изменений кода) Obsidian создаёт новый экземпляр модуля. Глобальные статические переменные внутри него не сохраняются между загрузками.

Минус: при выгрузке плагина кэш теряется. Но перестроить его после перезагрузки Obsidian — доли секунды.

## SQL-подобный язык для таблиц

Главная фича плагина — блок `fsrs-table`. Вы пишете в markdown-заметке:

````markdown
```fsrs-table
SELECT file as " ", difficulty as "D",
       stability as "S", retrievability as "R",
       date_format(due, '%d.%m.%Y') as "Следующее"
LIMIT 20
```
````


И получаете живую таблицу с карточками, которая автообновляется при повторениях.

<!-- ИЛЛЮСТРАЦИЯ 6: Анимация (GIF) — пользователь пишет SQL-запрос в fsrs-table, таблица появляется. Потом жмёт Again на карточке → таблица перерисовывается (D, S, R обновились, дата сдвинулась). Показать «живое» поведение. -->

Реализован полноценный парсер с нуля — лексер, парсер, AST (Abstract Syntax Tree — абстрактное синтаксическое дерево, которое представляет структуру запроса в виде, удобном для вычисления):

- **Лексер** — разбивает строку запроса на токены: ключевые слова, идентификаторы, числа, строки в двойных и одинарных кавычках, операторы
- **Парсер** — строит синтаксическое дерево из токенов. Поддерживает приоритет операторов: `AND` связывает сильнее чем `OR` (как в SQL)
- **Evaluator** — обходит AST и вычисляет условие для каждой карточки. Строковые поля (`state`, `file`, `due`) сравниваются лексикографически, числовые — математически

На момент написания поддерживается:

- `SELECT` — выбор полей и переименование через `AS`
- `WHERE` — условия со сравнениями (`=`, `!=`, `<`, `>`, `<=`, `>=`) и логическими операторами (`AND`, `OR`)
- `ORDER BY` — сортировка по возрастанию/убыванию
- `LIMIT` — ограничение количества строк
- `date_format()` — форматирование дат: `date_format(due, '%d.%m.%Y')`


## Тестирование без моков

Один из принципов разработки, который соблюдал в этом проекте: **нулевая толерантность к мокам**. Моки стороннего API (для плагина это сам Obsidian) запрещены — они создают иллюзию, что код работает, но без гарантий, что на реальном Obsidian всё не упадёт.

К тому же WASM-модуль — вещь в себе, без подвязок к файловой системе и внешнему окружению: принимает строку запроса и сырой YAML, отдаёт список, готовый к отображению. Замечательно покрываются тестами.

Вместо моков:

- **Rust-тесты (174 штуки)** — чистые функции, изолированные от окружения. Парсинг, вычисления, фильтрация — всё покрыто
- **TypeScript-тесты (57 штук)** — только чистые функции без Obsidian API: форматирование дат, извлечение имени файла, перевод состояний

Интеграционные тесты в Rust проверяют полный пайплайн: «парсинг SQL → добавление карточек в кэш → запрос → проверка результата».

## CI/CD: сборка, тесты, релиз одной кнопкой

Пайплайн в GitLab CI из семи стадий. Каждый коммит в `main` проходит:

1. `check` — `cargo fmt`, `cargo clippy`, `cargo test` (параллельно)
2. `build-wasm` — `wasm-pack build`, только если check зелёный
3. `encode-wasm` — встраивание WASM в base64 для бандла
4. `test` — TypeScript-тесты (vitest)
5. `lint` — `tsc --noEmit` + ESLint
6. `build` — финальная сборка `main.js`
7. `release` — автоматический релиз на GitHub с `main.js`, `manifest.json` и `styles.css`

Никаких ручных действий для релиза — тег на main'е автоматически создаёт релиз.

## Текущее состояние

Плагин уже можно брать и пользоваться. Базовый цикл работает стабильно: написал заметку, добавил `fsrs-table`, повторяешь карточки, таблица обновляется сама. SQL-фильтры реально удобны — например, можно вывести «только просроченные карточки сложностью выше 5» одной строчкой.

Самое эффектное — живые таблицы. Открываешь заметку с `fsrs-table`, жмёшь Again на карточке, и данные обновляются за тысячные доли секунды. А таблица перерисовывается с плавной анимацией. 

**Что уже готово:**

- Фильтрация и сортировка через SQL-подобный синтаксис
- Локализация на русский и английский (переключение в настройках Obsidian)
- CI/CD, который сам собирает и публикует релизы
