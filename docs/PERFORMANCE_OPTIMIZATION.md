# План оптимизации производительности FSRS плагина (черновик)

## Обзор проблемы
Текущая реализация плагина страдает от низкой производительности при:
1. Загрузке плагина - сканирование всех markdown файлов
2. Обновлении списка карточек - повторное сканирование всех файлов
3. Парсинге frontmatter - множественные вызовы WASM и парсинг YAML

## Пошаговый план оптимизации

### Шаг 1: Внедрение кэширования результатов сканирования
**Проблема**: Метод `getCardsForReview()` сканирует все markdown файлы при каждом вызове.

**Решение**:
1. Создать кэш FSRS карточек в памяти
2. Сохранять хэши файлов для отслеживания изменений
3. Обновлять кэш только при изменении файлов
4. Добавить TTL (время жизни) для кэша

**Ожидаемый выигрыш**: Ускорение последующих вызовов в 10-100 раз.

### Шаг 2: Инкрементальное обновление кэша
**Проблема**: При изменении одного файла пересканируются все файлы.

**Решение**:
1. Отслеживать события файловой системы Obsidian
2. Регистрировать обработчики для `create`, `modify`, `delete`, `rename`
3. Обновлять кэш только для измененных файлов
4. Использовать debounce для группировки событий

**Ожидаемый выигрыш**: Ускорение обновлений после изменений файлов.

### Шаг 3: Оптимизация парсинга frontmatter
**Проблема**: Для каждого файла происходит:
1. Чтение всего содержимого файла
2. Регулярное выражение для поиска frontmatter
3. Вызов WASM для парсинга (тяжелая операция)
4. Fallback парсинг на JavaScript

**Решение**:
1. Предварительная фильтрация по наличию полей FSRS в frontmatter
2. Кэширование результатов парсинга WASM
3. Упрощение регулярных выражений
4. Пропуск файлов без признаков FSRS полей

**Фактический выигрыш**: Ускорение парсинга более чем в 1000 раз (с ~950-1000 мс до ~0.7-0.8 мс).

### Шаг 4: Фоновая и ленивая загрузка
**Проблема**: Блокировка UI при сканировании большого хранилища.

**Решение**:
1. Разделить инициализацию на фазы:
   - Фаза 1: Быстрое сканирование метаданных
   - Фаза 2: Фоновый парсинг frontmatter
2. Ленивая загрузка дополнительных данных
3. Приоритизация видимых карточек
4. Progressive rendering интерфейса

**Ожидаемый выигрыш**: Улучшение UX, отсутствие "зависаний".

### Шаг 5: Оптимизация регулярных выражений
**Проблема**: Тяжелые регулярные выражения для парсинга YAML.

**Решение**:
1. Замена сложных regex на более простые паттерны
2. Использование позитивного просмотра вперед/назад только при необходимости
3. Компиляция regex один раз при инициализации
4. Кэширование результатов regex

**Ожидаемый выигрыш**: Ускорение парсинга на 10-20%.

### Шаг 6: Параллельная обработка файлов
**Проблема**: Последовательная обработка файлов блокирует выполнение.

**Решение**:
1. Использовать `Promise.all` с ограничением параллелизма
2. Батчинг файлов для обработки группами
3. Использование Web Workers для CPU-интенсивных операций
4. Приоритизация файлов с недавними изменениями

**Ожидаемый выигрыш**: Ускорение полного сканирования на 50-70%.

### Шаг 7: Оптимизация использования памяти
**Проблема**: Хранение полных объектов карточек и дублирование данных.

**Решение**:
1. Использовать lightweight структуры данных
2. Сериализация/десериализация только при необходимости
3. Очистка неиспользуемых данных из кэша
4. Использование WeakMap для временных данных

**Ожидаемый выигрыш**: Снижение потребления памяти на 30-50%.

### Шаг 8: Создание индекса FSRS файлов
**Проблема**: Линейный поиск карточек по всем файлам.

**Решение**:
1. Создать индексный файл `.fsrs-index.json`
2. Хранить метаданные карточек и пути к файлам
3. Обновлять индекс инкрементально
4. Использовать индекс для быстрого поиска

**Ожидаемый выигрыш**: Ускорение поиска карточек в 100+ раз.

### Шаг 9: Оптимизация WASM взаимодействия
**Проблема**: Множественные вызовы WASM для каждой карточки.

**Решение**:
1. Батчинг вызовов WASM - передача нескольких карточек за раз
2. Кэширование результатов вычислений WASM
3. Предварительная компиляция часто используемых функций
4. Использование SharedArrayBuffer для передачи данных

**Ожидаемый выигрыш**: Ускорение WASM операций на 40-60%.

### Шаг 10: Мониторинг и профилирование
**Проблема**: Отсутствие метрик производительности.

**Решение**:
1. Добавить timing для критических операций
2. Создать dashboard производительности в dev mode
3. Логировать медленные операции
4. Анализировать bottlenecks с помощью Chrome DevTools

**Ожидаемый выигрыш**: Возможность точной настройки производительности.

## Приоритетность выполнения
1. Шаг 1 (Кэширование) - максимальный эффект, минимальная сложность
2. Шаг 2 (Инкрементальное обновление) - улучшение UX
3. Шаг 3 (Оптимизация парсинга) - снижение CPU нагрузки
4. Шаг 4 (Фоновая загрузка) - улучшение воспринимаемой скорости
5. Шаги 5-10 - дополнительные оптимизации

## Метрики успеха
- Время загрузки плагина: < 1 секунда для 1000 файлов
- Время обновления списка карточек: < 100 мс после изменений
- Потребление памяти: < 50 МБ для 1000 карточек
- Отсутствие блокировки UI при операциях

## Инструменты для реализации
- `Performance.now()` для замеров времени
- `console.time()` / `console.timeEnd()` для профилирования
- Chrome DevTools Performance tab
- Obsidian API для событий файловой системы

## Рекомендации по тестированию
1. Тестировать на хранилищах разного размера (100, 1000, 10000 файлов)
2. Измерять производительность до и после каждой оптимизации
3. Проверять корректность работы после изменений
4. Тестировать edge cases (пустые файлы, битые YAML, большие файлы)

## Безопасность изменений
Каждый шаг должен быть:
1. Обратно совместимым с существующими данными
2. Изолированным для минимизации рисков
3. Протестированным на разных конфигурациях
4. С возможностью отката при проблемах
