# План миграции YAML структуры для FSRS плагина

## Обзор

Текущая реализация FSRS плагина хранит все параметры карточек непосредственно в frontmatter каждой заметки. Это приводит к избыточности данных и сложностям в обновлении алгоритмических параметров. Данный план описывает переход к разделенной архитектуре, где **индивидуальные данные карточки** хранятся в frontmatter, а **общие настройки алгоритма** выносятся в настройки плагина.

## Текущее состояние

### Текущий формат YAML
```yaml
fsrs_due: "2024-01-01T00:00:00Z"
fsrs_stability: 0.0
fsrs_difficulty: 0.0
fsrs_elapsed_days: 0
fsrs_scheduled_days: 0
fsrs_reps: 0
fsrs_lapses: 0
fsrs_state: "New"
fsrs_last_review: "2024-01-01T00:00:00Z"
```

### Проблемы текущего подхода
1. **Избыточность**: Каждая карточка содержит полный набор параметров алгоритма
2. **Сложность обновления**: Изменение параметров требует модификации всех карточек
3. **Несогласованность**: Разные карточки могут иметь разные версии параметров
4. **История отсутствует**: Нет возможности анализировать историю повторений

## Цели рефакторинга

### Основные цели
1. **Разделение данных**: Индивидуальные данные карточки vs. общие настройки алгоритма
2. **История повторений**: Сохранение полной истории сессий повторения
3. **Гибкость**: Легкое обновление алгоритмических параметров
4. **Обратная совместимость**: Поддержка существующих карточек

### Конкретные требования
- **В карточке хранить**: Историю повторений, индивидуальные флаги, ссылки на настройки
- **В настройках плагина хранить**: Параметры алгоритма FSRS, глобальные конфигурации
- **Миграция**: Автоматический переход от старого формата к новому

## Предлагаемая новая структура

### 1. Формат карточки (frontmatter заметки)
```yaml
---
srs:
  enabled: true
  settings_profile: "default"  # ссылка на профиль настроек
  review_sessions:
    - date: 2025-05-14T10:00:00
      rating: Good
      stability: 6.5
      difficulty: 3.2
    - date: 2025-05-07T09:00:00
      rating: Easy
      stability: 3.2
      difficulty: 3.0
  custom_parameters:
    # переопределения глобальных параметров (опционально)
    request_retention: 0.9
    maximum_interval: 365
---
```

### 2. Настройки плагина (в настройках плагина)
```typescript
interface FSRSSettingsProfile {
  id: string;                   // идентификатор профиля (default, custom1, etc.)
  name: string;                 // отображаемое имя профиля
  
  // Параметры алгоритма FSRS
  request_retention: number;    // целевой уровень запоминания (0.0-1.0)
  maximum_interval: number;     // максимальный интервал в днях
  w: number[];                  // веса алгоритма [17x1 вектор]
  enable_fuzz: boolean;         // включить случайное изменение интервалов
  
  // Глобальные настройки
  default_state: FSRSState;     // начальное состояние новых карточек
  notification_threshold: number; // порог уведомлений (дни)
  
  // Настройки отображения
  show_retrievability: boolean;
  show_hard_interval: boolean;
}
```

### 3. Вычисляемые поля (не хранятся, вычисляются на лету)
- **Текущая стабильность**: на основе последней сессии повторения
- **Текущая сложность**: на основе последней сессии повторения
- **Следующая дата повторения**: вычисляется из последней сессии и параметров алгоритма
- **Извлекаемость (retrievability)**: вычисляется на основе стабильности и времени

## Детали реализации

### 1. Утилиты для работы с форматами
```typescript
// Утилита для определения формата карточки
function detectFsrsFormat(frontmatter: any): 'legacy' | 'modern' | 'none';

// Конвертер из legacy в modern формат
function convertLegacyToModern(legacyCard: LegacyFSRSCard, settingsProfile: string): ModernFSRSCard;

// Парсер для modern формата
function parseModernFsrs(frontmatter: any, settings: FSRSSettings): ModernFSRSCard;
```

### 2. API для вычислений
```typescript
// Вычисление текущего состояния на основе истории
function computeCurrentState(reviewSessions: ReviewSession[], settings: FSRSSettings): {
  stability: number;
  difficulty: number;
  dueDate: Date;
  retrievability: number;
  state: FSRSState;
};

// Добавление новой сессии повторения
function addReviewSession(
  card: ModernFSRSCard,
  rating: FSRSRating,
  settings: FSRSSettings
): ModernFSRSCard;
```

### 3. Обработка смешанного режима
```typescript
async function getFsrsCard(file: TFile, plugin: FsrsPlugin): Promise<FsrsCard> {
  const frontmatter = await getFrontmatter(file);
  
  // 1. Пробуем modern формат
  if (hasModernFormat(frontmatter)) {
    return parseModernFsrs(frontmatter, plugin.settings);
  }
  
  // 2. Fallback на legacy формат
  if (hasLegacyFormat(frontmatter)) {
    const legacyCard = parseLegacyFsrs(frontmatter);
    return convertLegacyToModern(legacyCard, plugin.settings.defaultProfile);
  }
  
  // 3. Нет FSRS данных
  return null;
}
```

## Обратная совместимость

### Поддерживаемые сценарии
1. **Чтение legacy формата** (все версии)
2. **Автоматическая конвертация** при первом обновлении карточки
3. **Смешанное хранилище**: старые и новые карточки могут сосуществовать
4. **Постепенный переход**: пользователи могут мигрировать карточки по мере необходимости

### Миграционные команды
- **`fsrs:check-migration`**: Проверить карточки на готовность к миграции
- **`fsrs:preview-migration`**: Показать превью изменений без сохранения
- **`fsrs:migrate-single`**: Мигрировать текущую карточку
- **`fsrs:migrate-all`**: Мигрировать все карточки в хранилище
- **`fsrs:rollback-migration`**: Откатить последнюю миграцию (если есть backup)

## Заключение

Переход к разделенной архитектуре YAML позволит:
1. **Упростить обновление алгоритма**: Изменения в настройках применяются ко всем карточкам
2. **Сохранить историю**: Полная трассировка процесса обучения
3. **Повысить гибкость**: Разные профили настроек для разных типов карточек
4. **Улучшить аналитику**: Возможность анализа паттернов повторений

Миграция будет проводиться поэтапно с сохранением полной обратной совместимости и инструментами для безопасного перехода.
